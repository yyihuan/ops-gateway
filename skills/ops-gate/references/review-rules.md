# Review Rules

## 目标

这份规则给 `ops-gate` skill 使用。  
它的第一目标是兜底：让 agent 在做运维时，除基础只读动作外，其他动作都进入 plan + 审批流程。

## 1. 闸门外基础操作

只有同时满足以下条件，动作才可以视为闸门外基础操作：

- 不写文件，不删文件，不改权限，不改属主
- 不启停服务，不改系统状态
- 不访问远端可变资源
- 不带 `sudo`
- 不包含重定向到文件、`tee`、`mv`、`cp`、`rm`、`chmod`、`chown`、`systemctl`、包管理器、编辑器

常见例子：

- `pwd`
- `ls`
- `find`
- `cat`
- `sed -n`
- `head`
- `tail`
- `grep` / `rg`
- `stat`
- `df`
- `du`
- `uname`
- `id`

只要不确定，就默认进闸，不要默认放行。

## 2. 必须进闸的操作

以下任一情况成立，就必须进入 plan：

- 任何写文件、删文件、覆盖文件的动作
- 任何 `sudo` 或 root 相关动作
- 任何 service/process/network/package 变更
- 任何远端写入、远端执行、集群变更
- 任何会影响数据完整性、可用性、权限边界的动作

## 3. 分级建议

### Low

适用场景：

- 仍然是只读，但需要被审批者看见
- 例如一次性基线采集、只读诊断、环境核对

### Medium

适用场景：

- 范围明确、可控、局部可逆的写操作
- 例如在仓库 sandbox 下创建单个文件、更新单个非系统配置、生成局部缓存

要求：

- 明确 `effects.writes_paths`
- 明确 rollback 是否可信
- 如果存在覆盖风险，补 `step.backup`

### High

适用场景：

- 删除、覆盖、替换已有数据
- 修改 `/etc`、systemd、网络、权限
- 停服务、重启服务、改运行时配置
- 远端写操作、集群级变更

要求：

- 必须有清晰 `effects`
- 必须有明确人工审批对象
- 必须有 `step.backup`
- 必须给出可信 rollback，或明确写清无法回滚

### Critical

适用场景：

- 大范围删除
- 凭据、磁盘、分区、引导、核心网络、集群控制面变更
- 影响面大且失败代价高的动作

要求：

- 视为“默认不执行，除非范围和恢复路径都已被人工确认”

## 4. 审批展示最少要求

每条命令至少要让审批者看清：

- 它在读什么
- 它在写什么
- 它是否需要 root
- 它是否会影响服务或网络
- 如果失败，怎么回滚

## 5. Plan 作者规则

- 所有非基础操作都必须进 `steps[].commands`
- 不要把多个高风险动作揉成一个难以审的 step
- 尽量让 `review_note` 解释影响，而不是复述 shell
- 已知人工审批对象时，在 plan 顶层写 `operator`
